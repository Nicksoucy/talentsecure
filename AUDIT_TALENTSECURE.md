# Audit en profondeur — TalentSecure (XGuard)

> **Nature** : audit indépendant, **lecture seule**, multi-dimensions (architecture, sécurité, performance, données, fiabilité/DevOps, qualité de code, UX/produit, conformité Loi 25).
> **Date** : 2026-06-10 · **Périmètre** : monorepo `talentsecure` (backend Node/Express/Prisma→Neon, frontend React/Vite, déploiement Cloud Run).
> **Méthode** : lecture des 3 README + exploration par agents spécialisés en parallèle, vérification directe des constats critiques en `fichier:ligne`, typecheck + `npm audit` + sous-ensemble de tests sans base. **Aucune donnée ni aucun code applicatif n'a été modifié.**
> **Légende gravité** : 🔴 critique · 🟠 élevé · 🟡 moyen · 🟢 faible.

## 1. Résumé exécutif

**Santé globale : moyenne, avec une faille de sécurité critique qui domine tout le reste.** TalentSecure est un produit riche et déjà abouti (capture GHL, marketplace Stripe, uniformes, cartes géo, normalisation de villes soignée). Le backend a de vraies forces : validateur d'environnement *fail-fast*, webhook Stripe correctement signé avec *raw body* et idempotence, partage public de catalogue par token 256 bits, module uniformes bien découpé en services. Mais l'audit révèle une **chaîne d'accès non authentifié → exfiltration de toute la base de PII** qui doit être traitée avant toute autre chose.

**Top risques (à régler en priorité absolue) :**
1. 🔴 **Brèche d'autorisation majeure** : l'inscription client est **publique et auto-activée**, et un token CLIENT est accepté par les endpoints **admin** qui n'ont pas de garde de rôle. N'importe qui sur Internet peut s'inscrire, se connecter, puis lister **tous les candidats avec nom/email/téléphone/notes RH**, **tous les clients avec leur hash de mot de passe**, télécharger tous les CV et lire les catalogues d'autres clients.
2. 🔴 **Secrets de production commités** dans des fichiers `.md` suivis par git (URL Neon **avec mot de passe**, identique à la base live), sur un dépôt GitHub. + **token GHL en dur** dans 4 services.
3. 🔴 **Fuite de PII via les liens publics et les PDF de catalogue** (nom complet, n° BSP, n° de permis, notes d'évaluation internes, vidéo permanente) — enjeu Loi 25.
4. 🔴 **Extraction de compétences des CV cassée** (mauvaise API `pdf-parse` v2 + fichiers cherchés sur GCS/local au lieu de R2 + aucune extraction DOCX) : la fonctionnalité « Autre Compétence » tourne à vide et facture quand même l'IA.
5. 🔴 **Comptabilité de stock d'uniformes erronée** : chaque retour décompte −1 de trop → inventaire qui dérive et retours bloqués dès qu'une réserve atteint 0.
6. 🟠 **Observabilité nulle en prod** (le logger retire sa sortie console en production → rien n'arrive dans Cloud Logging) et **emails dupliqués** (cron sur chaque instance, dispatch non atomique).

**Quick wins (fort impact, effort S, faible risque) :** corriger les 2 crashs runtime frontend (import `CircularProgress`, signature `handleSelectAll`) ; `app.set('trust proxy', 1)` ; webhook GHL *fail-closed* ; sortir/`.gitignore` le CSV de PII ; corriger le filtre de doublon `OR:[{}]` du webhook ; révoquer le token GHL.

**Verdict** : aucune des fonctionnalités cœur n'est « pourrie », mais la **posture de sécurité et de confidentialité** n'est pas au niveau d'une plateforme qui manipule des renseignements personnels et de l'argent. Les correctifs critiques sont presque tous d'effort faible à moyen et peu risqués. Comptez **~1 semaine** pour purger les 🔴, puis un travail de fond plus étalé sur la dette (tests, découpe des pages géantes, migrations).

## 4. Ce qui va BIEN (forces réelles)

- **Paiement Stripe solide sur l'essentiel** : *raw body* monté avant `express.json`, signature vérifiée sans contournement, idempotence par contrainte unique, prix = `amount_total` réellement encaissé, révélation post-achat correctement *scopée* sur le client du token (`stripe-webhook.controller.ts`, `talent-marketplace.controller.ts:286-303`).
- **Anonymisation marketplace AVANT achat correcte** : `searchTalentsByCity` ne `select` que prénom/ville/province/notes ; la clé vidéo brute est retirée (`hasVideo` seulement). Le détail révèle nom/email/téléphone uniquement si un achat existe.
- **Démarrage *fail-fast*** : `config/env.ts` refuse de booter sans JWT secrets ≥ 32 c / DATABASE_URL et rejette les valeurs par défaut connues.
- **Protection SSRF du `cv-proxy`** : allowlist de domaines + re-vérification à chaque redirection — exemplaire.
- **Partage public de catalogue** : token 256 bits (`randomBytes(32).base64url`), expiration 30 j, rate-limit dédié.
- **Normalisation des villes** : `cityNormalize` (mojibake, Levenshtein, alias) + géocodage Nominatim QC-only en arrière-plan non bloquant + 1 marqueur/ville — travail récent soigné et cohérent.
- **Module uniformes (hors bug de stock)** : services mono-responsabilité, ledger append-only réconciliable, dettes figées par snapshot de coût, notifications avec `dedupKey`.
- **Frontend** : étanchéité admin/client réelle, init auth synchrone, lazy loading généralisé, `GeoPointsMap` bien factorisé, intercepteurs refresh *single-flight*.
- **Health checks** liveness/readiness séparés avec timeouts — niveau de maturité rare.
- **Codebase propre sur la dette explicite** : un seul `TODO` dans tout `src/` backend, `tsc --noEmit` backend à **0 erreur**.
